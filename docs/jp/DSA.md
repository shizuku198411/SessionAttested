# Development-Session Attestation (DSA)

[日本語](./DSA.md) | [English](../../DSA.md)


このドキュメントでは、`SessionAttested` の上位概念としての **Development-Session Attestation (DSA)** を定義します。

DSA は、リポジトリやバイナリ、ポートフォリオといった「成果物」だけでなく、**その成果物を生んだ開発セッション（開発過程）の証跡**も含めて評価・監査しようという考え方です。

`SessionAttested` は、DSA を実現するための実装アプローチの1つであり、dev container ワークフローを対象に、ホスト側でプロセス/ワークスペースの監査証跡を扱うことにフォーカスしています。

## 1. なぜ DSA が必要か（問題設定）

生成 AI を前提とした開発支援の普及により、実装速度は大きく向上しました。一方で、成果物だけを見た評価には、次のような限界が出てきています。

- 見た目の良い成果物が、そのまま実装理解度や説明能力を示すとは限らない
- 採用/試験/競技で禁止ツールが使われたかを確認しづらい
- IDE 拡張や周辺ツールの挙動を含め、開発セッション中に何が走っていたかを後から説明しづらい

DSA は、このギャップに対して **開発セッション/プロセス証跡**を追加することで補完する発想です。

## 2. DSA が扱うもの / 扱わないもの

DSA が扱うもの:

- 開発セッションに関する証跡を収集・検証するための考え方/仕組み
- 成果物中心の評価を、開発過程の証跡で補完する方法
- 監査・説明責任・スキル評価補助の基盤

DSA が直接は扱わないもの:

- コードレビュー、テスト、CI の代替
- 人の能力を自動スコアリングして断定する仕組み
- 監査対象外の環境まで含めた普遍的な行動証明

## 3. DSA で答えたい問い

DSA の仕組みは、たとえば次のような問いに答えやすくすることを目指します。

- この開発セッションでどのプロセスが実行されたか？
- どの実行体がワークスペースへ書き込んだか？
- どのファイルが変更され、どのプロセス系譜が関与したか？
- そのセッションはどの commit に紐づくか？
- ポリシー違反はあったか？
- 後から第三者が検証可能か？

## 4. DSA のレイヤ構造（実装を分けて考える）

実装・運用を整理する上では、DSA を次のレイヤに分けて考えると扱いやすいです。

1. **収集レイヤ（Collection）**
   - セッション証跡の収集（`exec`, workspace write, identities）
2. **紐づけレイヤ（Binding）**
   - セッション証跡と commit / repository の紐づけ
3. **検証レイヤ（Verification）**
   - ポリシー評価・署名検証
4. **レビューレイヤ（Review）**
   - 人間が確認できる形での提示（UI, レポート, Artifact）
5. **評価ポリシーレイヤ（Evaluation Policy）**
   - 収集した証跡をどう評価に使うか（禁止ツール、レビュー要件など）

`SessionAttested` は、特定の配置前提（監査主体管理下ホスト + dev container）において、主に 1〜4 を実装しています。

## 5. DSA のユースケース例

### 5.1 監査 / コンプライアンス

- 管理下セッションにおける禁止ツールの非観測確認
- 障害/インシデント時の不審プロセス実行の見直し
- 後から検証可能な説明用証跡の保持

### 5.2 開発プロセスの説明責任

- commit がどう作られたか（session -> process activity -> file changes）の説明
- 変更ファイルに禁止 exec 系譜が関与しているかの確認
- ポリシー fail 時の原因確認・レビュー

### 5.3 スキル評価の補助（成果物評価の補完）

- ポートフォリオの成果物評価に、開発過程の証跡を補助的に加える
- 採用/研修/試験で、特定ツール制限下の作業過程を確認する
- 教育用途で、最終解答だけでなく開発過程も見る

## 6. 既存ツールとの関係

DSA は、既存ツールの代替というより **補完レイヤ** として扱うのが適切です。

- EDR/XDR: 端末全体の可視性は強いが、開発セッション/commit 単位の説明には追加工夫が必要
- ネットワーク監視: 通信可視性は強いが、ファイル/プロセス/commit の説明には弱い
- CI/provenance: ビルド/配布の provenance には強いが、開発セッション provenance は対象外になりやすい
- コードレビュー/テスト: 成果物品質検証には強いが、開発過程の証跡は直接は扱わない

DSA は、これらに対して **開発セッションのプロセス証跡** を補います。

## 7. DSA における SessionAttested の位置づけ

`SessionAttested` は、DSA を実現する具体的な実装の1つです。

- ホスト側監査（LSM/eBPF）による dev container セッション観測
- 実行体 identity の集約（`exec`, `writer`）
- commit binding
- 署名付き証明 + verify
- 人間レビュー向け成果物（`ATTESTED_SUMMARY`, `ATTESTED_WORKSPACE_OBSERVED`, WebUI）

このため `SessionAttested` は、

- 「AI Agent 検知専用ツール」ではなく
- **DSA（開発セッション証明）を実現するための証跡/検証基盤**

として捉えるのが本質に近いです。

## 8. 限界と運用設計（重要）

DSA を使う上では、実装の外側で決めるべき事項があります。

- どのポリシーを「許容/禁止」とするか
- どの証跡を保持・共有するか
- プライバシー境界をどう設計するか
- 採用/教育/監査で証跡をどう解釈するか

DSA は証跡収集・検証の仕組みを提供しますが、評価基準そのものは組織/チーム側の設計が必要です。
