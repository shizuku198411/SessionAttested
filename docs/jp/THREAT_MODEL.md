# Threat Model

[日本語](./THREAT_MODEL.md) | [English](../../THREAT_MODEL.md)


`SessionAttested` は、監査主体が管理するホスト上の開発セッションに対して、`exec` / `workspace write` の監査証跡を収集し、ポリシーに基づく検証結果を署名付きで扱うための仕組みです。

このドキュメントは、PoC 時点での脅威モデル（何を守るか、何を前提とするか、何は守らないか）を整理します。

## 1. 守りたいもの（Security Goals）

- 開発セッション内で実行されたプロセス（`exec`）の監査可能性
- ワークスペースへの書き込み主体（`writer`）の監査可能性
- セッション監査結果と commit の紐づけ（commit binding）
- 監査結果（署名付き証明）の署名検証可能性
- ポリシー違反（例: 禁止実行体の起動）の検証可能性

## 2. 主張できること（What It Proves）

PoC 時点での主張は、次のように表現するのが適切です。

- 監査主体の管理下にあるホストで収集された当該セッション監査証跡に基づく限り、
  ポリシーで禁止された実行体/書き込み主体が観測されたかどうかを検証できる

特に AI Agent 利用可否の文脈では:

- 「禁止対象の Agent 実行体が監査対象セッション内で観測されたか」を検証できる

## 3. 主張しないこと（Non-Goals / Limits）

PoC 時点では、以下は直接は証明しません。

- 被監査主体が別環境で生成した成果物を持ち込んでいないこと
- 監査対象外の環境で AI/自動化ツールを使っていないこと
- ホスト管理者権限を持つ攻撃者が不正を行っていないこと
- カーネル/eBPF/コンテナ基盤が完全に信頼できることを超える保証

「絶対的な不使用証明」ではなく、管理下セッションに対する高信頼な監査証跡・検証フローです。

## 4. 信頼前提（Trust Assumptions）

PoC では、少なくとも以下を信頼前提とします。

- 監査主体が管理するホスト
- ホスト OS / カーネル / eBPF 実行基盤
- `attested` バイナリ（監査主体が意図したビルド）
- 署名鍵の管理（`attested attest` で使う秘密鍵）
- 監査対象コンテナの起動方法（意図した mount / config）

## 5. 想定する攻撃・回避（Examples）

### 5.1 検知しやすいもの

- 禁止された実行体の起動（`forbidden_exec`）
- 禁止された書き込み主体の workspace 書き込み（`forbidden_writers`）
- 監査結果 JSON の事後改ざん（署名検証で検出）

### 5.2 検知しにくい/スコープ外のもの

- 監査対象外の別環境で作成したコードの持ち込み
- ホスト管理者によるログ/collector/カーネル層改ざん
- `exec` を伴わない外部要因によるファイル内容変更（設計外ケース）

## 6. 判定上の考え方（PoC 運用）

PoC 時点では、判定の重みづけは以下を推奨します。

- `forbidden_exec`: 主判定
- `forbidden_writers`: 補助判定

理由:

- 実際の書き込み syscall を発行する主体は、上位ツール本体ではなく `bash` / `node` / `python` 等に委譲される場合がある
- 一方、禁止ツール本体の起動は `exec` 側で比較的安定して観測しやすい

## 7. 残余リスク（Residual Risks）

- 実行体 identity 解決失敗（`identity_unresolved`）が一定数発生する可能性
- ツール更新により path / hash が変化し、ポリシー保守が必要になる
- 監査対象コンテナ構成（UID/GID, mount, SSH, editor 実装）により見え方が変化する

これらは運用上、

- `audit_summary.json` の未解決件数確認
- `attested policy candidates` による候補抽出
- ポリシーの定期更新

で管理します。

## 8. 強化候補

- 透明性ログ（transparency log）への署名付き証明の登録
- KMS/HSM による署名鍵管理
- タイムスタンプ局（TSA）連携
- `write` と `exec` の lineage 相関強化（親子プロセス連鎖の明示）
- 実行時 deny モード（LSM ベースの拒否制御）
